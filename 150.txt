5.今までのプロジェクトで一番難しかった問題は何ですか。それはどのように解決しましたか。

SDK の API レスポンスをそのまま DTO に変換した結果、テスト時に大量の型エラーや構造不整合が発生しました。

現在の案件における課題点は、フロントエンドからBFFのAPIを呼び出し、その後さらにSDKのAPIを経由して他システムと連携し、レスポンス結果を処理した上でDTOに変換し、BFFのAPIレスポンスとしてフロントエンドに返却する必要がある点です。
複雑なデータ構造のマッピングや、TypeScriptによる厳格な型チェック、さらに100%のテストカバレッジを実現することが難しいポイントとなっています。

カバレッジテストにおいては、正常系だけでなく、条件分岐やエラー処理などの部分も100％テストカバレッジを達成する必要があり、テストケースの作成に多くの時間がかかる点が課題となっています。

NestJs进行dto变换的注意点
NestJSでデータ変換を行う際、SDK APIのレスポンスデータをDTO変換する前にはいくつか注意点があります。
特に、レスポンスデータの中にJSON形式の文字列が含まれている場合、そのままDTOに変換しようとすると型エラーや意図しないデータ構造になる可能性があります。
そのため、まずJSON文字列に対応した型定義のDTOファイルを作成し、正しい型に整えてから、外側のオブジェクトと統合してDTO変換を行う必要があります。
この手順を踏むことで、型安全かつ意図したデータ構造でサービス層やコントローラーに渡すことができ、後続の処理もスムーズに進められます。


6.ソースを書く時/コーディングする時、注意すべきこと/心掛けているのは何ですか。　
1. インターフェース規約の統一
リクエスト方式（GET/POST/PUT/DELETEなど）、パス、パラメータ形式、レスポンス構造を明確に定義する。
RESTfulやGraphQLなどの規約を推奨し、インターフェース名や内容を分かりやすくする。
2. データ型とバリデーション
前後でデータ型（string、number、boolean、配列、オブジェクトなど）を事前に取り決め、型不一致によるエラーを防ぐ。
バックエンドは入力値の厳格なバリデーションを行い、フロントエンドも基本的なチェックを実施する。
3. エラー処理の仕組み
バックエンドは統一されたエラーコードとメッセージを返し、フロントエンドはそれに応じて適切な処理（ダイアログ表示、リダイレクト、再試行など）を行う。
標準的なエラーレスポンス形式（例：{ code: number, message: string }）を取り決める。
4. セキュリティ
パラメータチェック、SQLインジェクション、XSS、CSRFなどのセキュリティ対策を徹底する。
機密性の高いインターフェースは認証（JWT、セッション、OAuthなど）を必ず実装する。
5. インターフェースドキュメント
Swagger、OpenAPI、Postmanなどのツールでインターフェースドキュメントを管理し、開発・テスト・連携を円滑にする。
ドキュメントは常に最新状態に保つ。
6. パフォーマンスと拡張性
ページング、フィルタ、ソートなどを考慮し、一度に大量データを返さない設計にする。
バッチ処理や非同期処理も検討し、パフォーマンスとユーザー体験を向上させる。
7. バージョン管理
インターフェース変更時はバージョン管理（例：/api/v1/xxx）を行い、既存クライアントへの影響を最小限にする。
8. クロスオリジンとCORS
バックエンドでCORS設定を正しく行い、フロントエンドから安全にアクセスできるようにする。
まとめ：
インターフェース規約、データ型、エラー処理、セキュリティ、ドキュメント、パフォーマンス、バージョン管理、CORS設定が前後端インターフェース設計の主な注意点です。

125. AWSでプロジェクトをリリースするときどんな技術を使用しましたか。
ローカル環境から GitHub へ git push を実行すると、GitHub 側で検知された Webhook イベントが AWS CodeConnections の接続を伝達され、これに連動して CodePipeline が自動的にパイプラインを起動します。

Source および Build ステージ：CodePipeline が最新のソースコードを自動的にダウンロードし、CodeBuild を起動します。CodeBuild は特権モード（Privileged）環境で docker build を実行し、
ビルドされた最新のイメージを Amazon ECR にプッシュすると同時に、デプロイスクリプトを含むアーティファクト（展開マニフェスト）を出力します。

Deploy ステージ：CodePipeline がデプロイ情報を CodeDeploy サービスに引き渡します。

EC2 インスタンスでの実行：EC2 上で稼働する CodeDeploy Agent がクラウドからのデプロイ指示を検知し、deploy.sh スクリプトを自動的にダウンロードしてローカル環境で安全に実行します。
このスクリプトは AWS 認証を通じて ECR にログインし、旧コンテナの停止、ECR からの最新イメージのプル、およびアプリケーションのシームレスな再起動を行います。


126.EC2 / RDS / S3 / EKS/ ECRはそれぞれどんなサービスでしょうか。
EC2（イーシーツー）「AWSの中に作れる、自分専用のレンタルパソコン（サーバー）だよ」。
RDS（アールディーエス）「面倒な設定や管理を全部AWSがやってくれる、データ保存用の箱（データベース）だよ」
S3（エススリー）  「容量無限で、めちゃくちゃ安くて壊れない、クラウド上の特大ファイル置き場だよ」
EKS（イーケーエス）「たくさんのDockerコンテナを、自動でいい感じに管理・運用してくれるサービスだよ」もっと詳しく： コンテナを管理する「Kubernetes（クバネティス）」という難しい仕組みを、AWS上で簡単に使えるようにしたサービス。大規模なシステムでコンテナが何十個、何百個になっても自動でスケールさせてくれます。
ECR（イーシーアール）「自分たちのプロジェクトで作ったDockerイメージを、安全に保管しておく専用の倉庫だよ」もっと詳しく： さっきのCodePipelineの会話でも出てきた、Dockerイメージのプライベートな置き場です。ここにイメージを預けておいて、EC2やEKSがそれを引き抜いて（Pullして）アプリを起動します。

128. AWSの設計と構築の経験がありますでしょうか

はい、AWSの設計および構築の経験は豊富にあります。

小規模なWebアプリケーションのインフラ立ち上げから、可用性（システムが落ちにくい構成）を高めた本格的なエンタープライズ向けのアーキテクチャ設計まで、要件定義、構築、自動化まで一連のフェーズに対応可能です。

具体的には、以下のような設計・構築・運用の経験とスキルセットを持っています。

🏗️ 1. 安全で堅牢なネットワークインフラ設計（VPC/セキュリティ）
安全なネットワーク分割：パブリックサブネットとプライベートサブネットを明確に分け、データベース（RDS）やコンテナ基盤（EKS/ECS）などの基幹系をプライベートスペースに完全隔離する設計。

セキュアなアクセス：NAT Gateway を経由した安全なアウトバウンド通信の確保、Internet Gateway を用いたパブリックアクセスの制御、および Security Group とネットワークACL（NACL）による最小権限のトラフィック制御。

🔄 2. 高可用性と負荷分散（マルチAZ / スケーリング）
ダウンタイムの最小化：Application Load Balancer (ALB) をフロントに配置し、複数のアベイラビリティゾーン（マルチAZ）にEC2やコンテナを分散配置する、単一障害点（SPOF）のない設計。

自動スケール：CPU負荷やリクエスト数に応じてサーバー数を自動で増減させる Auto Scaling の設計・導入。

🚢 3. 近年のモダンなクラウドネイティブアーキテクチャ（コンテナ・サーバーレス）
コンテナ運用基盤：Amazon ECS (Fargate) や Amazon EKS を用いた、OS管理フリーで拡張性の高いコンテナオーケストレーション環境の設計・構築。

サーバーレス：AWS Lambda、Amazon API Gateway、Amazon DynamoDB を組み合わせた、サーバーの常時起動コストをゼロにするイベント駆動型システムの構築。

⚙️ 4. インフラのコード化（IaC）と自動化
Terraform / AWS CloudFormation：コンテナ、VPC、IAMなどのインフラ一式をコード化し、ボタン一つで同じ環境を「ステージング用」「本番用」とクローンできる環境の構築。

CI/CD連携：今回のテーマにもあった AWS CodePipeline / CodeBuild / CodeDeploy や GitHub Actions を用い、GitHub への git push から ECR へのイメージ登録、ECS/EC2 へのゼロダウンタイム（藍緑）デプロイまでの完全自動化。

セール開始時のWebフック通知・クーポン配布

また、データベースや外部APIの認証情報といった機密データ（Credentials）を扱う際は、セキュリティベストプラクティスに従い、Lambda の環境変数に直接書くのではなく、

マイクロサービス化された環境だったため、どのコンポーネントがボトルネックになっているかを特定するために、まずは AWS X-Rayエックスレイ のサービスマップとトレース機能を使用しました。

X-Rayのセグメントタイムラインを分析したところ、ECSコンテナ側の処理は数十ミリ秒で終わっているものの、**RDS Auroraへのクエリ実行部分（SQL呼び出し）に8秒以上かかっている（タイムラインが異常に伸びている）**ことが一目で判明しました。

バグ対応として、
アプリ側のクエリの書き方をリファクタリングすると同時に、DBに適切な複合インデックス（Composite Index）を追加しました。また、今後の再発防止策として、3秒以上かかるクエリが発生した際にCloudWatchからSlackへ警告通知が飛ぶアラートの閾値設計も行いました。

このように、CloudWatchとX-Rayを組み合わせることで、複雑な分散システムであっても、勘に頼らず「データ（ログとトレース）」に基づいて迅速に原因を特定し、修正することができました。
