少人数（1 ユーザー / 1 ループ）でテストを実行し、エラーなくリクエストが飛ぶか確認。

123.テスト結果のエビデンスのレビューはされていたか。その時やり直しはあったか。どんな指摘をよく受けたか。
1. 事前状態（Before）の取得漏れ
指摘内容: 「処理実行後（After）のDBキャプチャや画面はあるが、実行前（Before）の状態が残っていないため、今回のテスト操作によって正しくデータが変化したのか（元からそのデータだったのではないか）が証明できない」

やり直しの理由: データの状態変化（UPDATEやDELETE）を伴うテストにおいて、比較対象がないため。

改善策: テスト開始直後に必ずDBの SELECT 結果や一覧画面の初期状態をスクショ・保存する癖をつけました。


2. 異常系テストにおける「裏側の確認」漏れ
指摘内容: 「画面上に『システムエラー』のメッセージが出ているエビデンスはあるが、裏側のDBに不正なデータが登録されていないこと（ロールバックされたこと）のエビデンスがない」

やり直しの理由: 表面上はエラーに見えても、裏側でデータ不整合が起きている（サイレント障害）可能性を払拭できないため。

改善策: 異常系のテスト（エラーハンドリング）を成功とするには、「画面のエラー表示」＋「DBやサーバーログの無事（変更なし）」をセットで証明するよう意識を変えました。

133.データ移行する際、何か障害が発生したことがありますか。

本番データの「イレギュラー（外れ値）」による移行バッチ停止
旧システムから新システムへ顧客データをバッチ処理で移送・変換（ETL）していた際、想定外のデータが含まれており、移行バッチが途中でクラッシュ（異常終了）しました。

原因:
旧システムのデータベース上、通常は入力されるはずのない NULL や制御文字、超ロング文字列（仕様外の文字数）が過去の運用データ内に残存していたため、新システムのバリデーションチェックに引っかかりました。

影響:
テスト環境のダミーデータ（きれいに整理されたデータ）では正常終了していたため、本番移行の当日になって初めて発覚し、移行作業時間が大幅に延びました。

対策（教訓）:

本番移行の前に、本番データを個人情報マスキングした上で「事前リハーサル（ドライラン）」を必ず実施し、データの全件洗い出しとクレンジング（事前補正）を行う運用に変えました。

移行スクリプト側も「1件エラーが起きても全体を止めず、エラーログを出力してスキップ（またはデッドレターキューへ退避）し、後から個別リトライできる構造」に修正しました。


文字コード変換・文字化けによるデータ破損
旧システムの文字コード（Shift-JIS / Windows-31J）から新システム（UTF-8）へデータをコンバートする際、外字や特殊文字（旧字体・環境依存文字）が文字化け（? や  に置換）しました。

移行前に旧DB内で「環境依存文字・非ASCII文字」が含まれるレコードを抽出する SQL スクリプトを作成し、事前に変換ルールを定義・テストするようにしました。




134.データ移行する際、データを読んでいる先を変えないといけないと思います。その辺対応したことありますか。


一括切り替え（Big Bang Approach / メンテ時間を設ける場合）
夜間や休日にシステムを一時停止（メンテナンス状態）にし、データの最終同期を行った上で一気に参照先を切り替える最も標準的な手法です。

切り替え方法:

DNS / ロードバランサー（ALB等）の切り替え: ユーザーからのドメインアクセス先（例: api.example.com）の向き先を、旧環境のIPから新環境のIPに変更。

環境変数 / 設定ファイルの書き換え: アプリケーション側の DB 接続文字列（JDBC URL やホスト名）を書き換えて再起動。


段階的切り替え / 2重書き込み（Dual Write / メンテ時間を最小化・無停止にする場合）
無停止（Zero Downtime）または短時間の停止で移行したい場合に用いる高度な手法です。

 [ ユーザー ]
     │
     ▼
 [ 新API / Gateway ] ───(1. 参照は新DBへ)───► [ 新 DB ]
     │
     └───(2. 更新は両方へ同期/2重書き)───► [ 旧 DB ] (切り戻し担保用)
切り替え方法:

旧DB ➔ 新DB へのリアルタイムレプリケーション（CDC: Change Data Capture）: AWS DMS（Database Migration Service）や Debezium 等を使い、旧DBの変更差分をリアルタイムで新DBへ流し込む。

アプリケーション側でのスイッチ制御: フィーチャーフラグ（Feature Flag / 設定値）を用いて、「特定のユーザー群だけ参照先を新DBに変える」といったカナリアリリース（段階切り替え）を実施。

137.今までSQLのパフォーマンス（性能面）チューニングのご経験があるか。

はい、SQLおよびデータベースのパフォーマンス（性能）チューニングの実務経験があります。

スロークエリ（処理に時間がかかっているSQL）の特定から、実行計画の解析、適切なインデックス（Index）の設計、クエリのリファクタリングまで、一通り対応してきました。


ボトルネックの特定:

スロークエリログ（Slow Query Log） や APM（Datadog, New Relic 等） を監視し、応答時間が遅い（または実行頻度が高くDB負荷になっている）SQLを特定。

実行計画の解析:

EXPLAIN や EXPLAIN ANALYZE コマンドを実行し、データベースがどのようにデータを検索しているか（フルテーブルスキャンか、インデックス検索か、結合順序は適切か）を確認。

対策の実施と検証:

インデックス追加、SQL書き換え、テーブル設計（正規化/非正規化）の検討を行い、修正前後で実行速度（レイテンシ）と読み込みレコード数が削減されたかを検証。

142.ETLという技術はご存知でしょうか。

はい、ETL（Extract / Transform / Load）はよく知っており、データ移行やデータ分析基盤（データウェハウス：DWH）の構築などで深く関わるコア技術です。

システム間のデータ連携や、旧システムから新システムへデータを移送・加工して格納するプロセスにおいて中心的な役割を果たします。


[データソース]               [ETLプロセス]               [ターゲット]
 (旧DB / CSV / API) ──►  E ──► T ──► L  ──► (新DB / DWH / BI)


Extract（抽出）:

様々なデータソース（RDB、NoSQL、CSV/TSVファイル、外部Web API、ログファイルなど）から、必要なデータを取得する処理。

Transform（加工・変換）:

抽出したデータを、移行先・保存先のフォーマットに合わせて変換する処理。

主な処理: データ型の変換、文字コード変更（Shift-JIS ➔ UTF-8）、不要データのクレンジング（NULL補正や不要行の除外）、テーブル結合・集計、名寄せなど。

Load（書き込み・格納）:

加工後のデータを、ターゲット（新システムのDB、データウェアハウス（Snowflake / BigQuery 等）、データレイク）へ書き込む処理。




本番データ特有の「外れ値」対策（Transform段階）:

旧DBに存在する想定外の型・文字数・NULLデータなどにより、TransformやLoadの段階で処理がクラッシュしないよう、不整合データの検知（バリデーション）とエラーログ化（スキップ処理）を設計します。

大量データ挿入の高速化（Load段階）:

数百万件規模のLoadを行う場合、1件ずつ INSERT すると膨大な時間がかかります。そのため、バルクインサート（Bulk Insert / Copy コマンド）の利用や、一時的なインデックス・制約の無効化（Disable/Drop）を行ってパフォーマンスを最大化します。

ETLツールの活用:

Python（Pandas）や SQL（Stored Procedure）等でスクリプトを自作するケースもあれば、AWS Glue, Talend, Apache NiFi, Informatica などの専用ETL/ELTツールを活用してパイプラインを自動構築・監視するケースもあります。

146.要件定義を担当した経験ありますか。要件定義でどんな様な書類を作成しましたか。

要件定義は、発注者（顧客・業務部門）の「実現したいこと・課題」を吸い出し、システムとして「何を（What）構築するのか」を明確にして合意形成を図る非常に重要な工程です。

1. 業務要件（Business Requirements）に関する書類
業務の「現状（As-Is）」と「理想（To-Be）」を整理し、新システムでどのように業務が変わるかを提示するドキュメントです。

業務フロー図（As-Is / To-Be 業務フロー）:

業務の流れ、関係する部署・担当者、新システムとの境界を可視化したフロー図（BPMNやスイムレーン形式）。

業務要件定義書:

システム化の目的、解決すべき課題、期待される効果（KPI）、業務上のルールや制限事項をまとめた文書。

2. 機能要件（Functional Requirements）に関する書類
システムが具体的に「どのような機能を提供するか」を定義するドキュメントです。

機能一覧表（機能要件定義書）:

システムが持つべき全機能を洗い出し、大・中・小分類、優先度（Must / Should / Want）、概要を一覧化（WBSや進捗管理のベースにもなる重要書類）。

画面一覧・画面遷移図・ワイヤーフレーム（画面イメージ）:

画面のレイアウト（ラフ案）や画面間の遷移関係を示し、顧客と具体的なUI/UXのイメージを共有・合意するための資料。

ユースケース記述書 / ユーザーカンプ・ユーザーストーリー:

「誰が（アクター）」「どのような目的で」「どんな操作・処理を行うか」をシナリオ形式で整理した書類。

データモデル（概念データモデル / 概念ER図）:

新システムで扱う主要な情報（ユーザー、注文、商品、決済など）とその相互関係（リレーション）を整理した図。

外部システム連携要件定義書:

既存システムや外部サービス（決済API、会計システムなど）との連携方式（API、ファイル連携、バッチなど）、連携データ項目、頻度を定義した資料。
